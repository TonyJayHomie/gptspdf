```Issue 1 — PolaritySign() always returns 1f.
Not a bug for this use case. AC, iRacing, and Forza encode force direction in the sign of the Magnitude field of the Constant Force report, not in the polar angle. Single-axis wheel, Magnitude carries the sign, PolaritySign() returning 1f is fine. Would be wrong for multi-axis joystick builds.
Issue 2 — Inertia uses velocity, not acceleration.
EvaluateCondition for EffectType.Inertia uses axisVelocity as the input. DirectInput inertia is supposed to resist changes in velocity (i.e., proportional to acceleration = dV/dt). Using velocity instead of acceleration means it behaves more like a damper than an inertia effect. This is a known approximation in most DIY FFB implementations. Since condition effects are disabled by default and inertia is rarely used in sim racing, this is low priority. Just worth knowing when you eventually enable conditions.
Issue 3 — Main PC app not in this zip.
BridgeCore.cs is the engine. BridgeCoreTests.cs tests it. But the file that actually hooks into vJoy (FfbRegisterGenCB, reads PT_CONSTREP packets, calls FfbEngine.Handle(), then calls UnoProtocol.TorqueCommand() and writes to serial) — that's not here. That's the FFBBridge_v3.cs equivalent for v4, and it needs to be written or updated. The v3 file needs these changes to work with v4:```

```Executive Summary
The proposed FFB-to-VESC bridge links a PC game’s Force-Feedback (FFB) output to a VESC-controlled hoverboard motor via an Arduino Uno. We verified each part of this chain against official documentation. The Arduino Uno R3 (16 MHz ATmega328P, CH340 USB interface
) uses digital pins 10/11 for SoftwareSerial to the VESC. A Flipsky “UART adapter” (a simple 5V TTL-serial interface) connects UNO TX/RX to the FSESC UART lines. The Flipsky FSESC 4.20 (Dual) ESC is powered by 24 V on its XT60 connectors (observe correct polarity
) and drives a 3-phase hoverboard BLDC motor. Motor phase leads attach to the three output pads (order can be swapped if rotation is reversed
). The ESC’s 6-pin hall connector expects +5V (red) and GND (black) from the motor, with the three sensor outputs (typically yellow, green, blue) going to H1, H2, H3
. In VESC-Tool, the motor is set to FOC mode with hall sensors (as recommended for smooth operation
) and the detection wizard is run. UART is enabled at 19200 baud to match the Arduino. The motor current limits are set low (≈15 A) well under the ESC’s 50 A continuous rating
.

The Arduino sketch uses Serial (115200 baud USB) to read float values via Serial.parseFloat() from the PC, then sends them via SoftwareSerial (19200 baud) using the SolidGeek VescUart library (setCurrent() and periodic sendKeepalive()). We note parseFloat’s quirks (e.g. returning 0.00 on stray characters
) and SoftwareSerial’s limitations at higher baud
. The PC-side C# app uses vJoy FFB callbacks to capture the constant-force effect magnitude (±10000 range
) and scales it to ±15 A, appending a newline before sending. The COM port (e.g. “COM3” at 115200 baud) must match the Uno’s USB connection. Throughout, we estimate end-to-end latency: ~1–5 ms USB+PC processing, ~5 ms at 19200 baud, plus software overhead, for a total on the order of 5–15 ms. The keepalive is sent every 50 ms (more frequent than typical 1 s
, ensuring the VESC never times out).

Safety precautions include the Arduino’s built-in clamping (constrain(-15,15)), the ESC’s over-current protection, proper fusing on the 24 V supply, and thermal monitoring of the ESC and motor. Below we detail each segment, cite relevant specs, and summarize changes/improvements.

Hardware Connections Verification
Arduino Uno R3 (CH340 clone) – This 16 MHz ATmega328P board has 14 digital I/O (6 PWM) and 6 analog inputs
. Its USB-to-serial chip is a CH340 instead of the original ATmega16U2
. It is correct to power the Flipsky adapter from the UNO’s 5 V pin (VIN) and GND. In the sketch, SoftwareSerial vescSerial(10,11) uses UNO pins 10 (RX) and 11 (TX). These pins are valid for SoftwareSerial, though at high baud (19200) we note from Arduino forums that 38400 baud “can work” and that TX is more reliable than RX
. Using 19200 baud is near the practical limit, so interrupts and CPU load may cause occasional jitter. A recommended improvement is to use the hardware UART (Serial1 on a Mega or AltSoftSerial) if available, or lower the baud.

Flipsky UART Adapter – This 4-pin module bridges TTL serial to the VESC. The wire labels typically are: GND, VIN (5 V), VTX, VRX. In practice the adapter’s “VTX” is its TX output, connected to the Uno’s RX (pin 10), and “VRX” is its RX input, connected to the Uno’s TX (pin 11). GND must be common, and VIN should be tied to the Uno’s 5 V to power the adapter. (This matches standard FTDI-style pinouts.)

FSESC 4.20 ESC (Dual) – The Flipsky 4.20 Plus ESC uses XT60 bullet connectors for battery power. It supports 8–60 V (3–13S LiPo)
. The instructions correctly route +24 V to XT60 positive and 0 V to negative; polarity must be checked
. The ESC’s three motor-phase outputs (A/B/C) connect to the hoverboard motor’s three thick wires. Order doesn’t matter for function, but if rotation is reversed, swapping any two phases will correct direction
.

The ESC has a 6-pin hall sensor port. The hoverboard motor’s hall wires go here: +5 V (red), GND (black), and the three sensor outputs. Commonly these are yellow, green, blue
. For example, one hoverboard motor’s wiring is Red=+5V, Black=GND, Yellow=Hall 1 (A), Green=Hall 2 (B), Blue=Hall 3 (Z)
. (Labels H1/H2/H3 on the ESC should correspond, but the wizard will sort them.) Ensuring correct connections is critical: the Flipsky manual emphasizes “phase wires are correctly matched” and “sensor wires properly connected” or the motor will behave erratically
.

Figure: Example Arduino-to-BLDC controller wiring (PWM-based ESC example) – Arduino is powered by USB (5 V/GND) and uses digital pins (e.g. 10/11) for TX/RX to the ESC’s control board. In our FFB-bridge, the Flipsky TTL adapter plays the ESC’s control board role.

Summary: All listed connections match standard practice. The Uno’s CH340 clone has the expected pinout
. The Flipsky adapter’s pinout (GND, VIN, TX, RX) is consistent with wiring to Arduino RX/TX. The ESC’s XT60 and hall connectors are wired per Flipsky guidelines
. If direction or sensor phasing is wrong, swapping two phase wires or adjusting the hall order is recommended (the VESC wizard handles inversion if needed).

VESC Tool Configuration
In VESC-Tool, the motor is configured as FOC (Field-Oriented Control). This is correct for smooth, low-noise operation with hall sensors
. The user must run the FOC detection wizard with the motor and halls connected. The VESC-Tool wizard explicitly states “we will run the motor in FOC mode and make use of Hall sensors”
.

Key settings to verify:

UART: Under App Settings → UART, baud rate must be set to 19200 and enabled. The Arduino code opens SoftwareSerial at 19200, so they must match. Some Flipsky documentation warns this is critical.
Current Limits: The sketch and documentation suggest ~15 A. The ESC’s spec shows 50 A continuous (single)
, so 15 A is safely below the ESC’s capability. However, the motor’s actual rating should be checked (many hoverboard motors run ~10–20 A continuous). By default, VESC Tool may have much higher limits; these should be reduced. The manuals note “High motor current” protection
, so configure Motor Settings → Current to 15 A. Also set Battery Current to safe limits.
Detection: Run the detection wizard carefully with rotor locked. For FOC with halls, after detection the tool shows hall timing (λ) applied
. Confirm “everything spinning in the right direction” and invert if not
.
Missing/Unsafe Defaults: Check that the battery cutoff voltage matches your 24 V source, since the wizard defaults can be LiPo-specific (3.4 V/cell start cutoff
). Also ensure Current max (duty cycle, brake, etc.) are set low enough; VESC warns if left at 85% default in skates applications. Any unused features (e.g. braking) should be disabled or set to 0.
In summary, the VESC Tool should be configured for BLDC FOC mode with halls and UART at 19200. The provided instructions align with Flipsky’s guidance
.

Arduino Uno Sketch Review
The Uno sketch (shown in the user file) is mostly correct but has areas for robustness:

Serial Ports: The Uno uses Serial (hardware) at 115200 baud for PC USB, and SoftwareSerial vescSerial(10, 11) at 19200 baud for the VESC link. Pin 10 as RX and pin 11 as TX is allowed
. We note from Arduino docs that SoftwareSerial is inherently slower and less reliable, especially on RX at high speeds
. If communication errors occur, consider using hardware serial (e.g. Mega’s Serial1) or AltSoftSerial for better timing.

Parsing Input: The code uses Serial.parseFloat() to read the incoming current command. In practice, parseFloat() skips non-numeric characters and stops at the first non-digit or minus. As one Arduino forum explains, if a stray newline or non-numeric arrives, parseFloat() will return 0.00
. For example, if the PC sends “12.34\n”, the newline can cause the next call to yield 0.00. It’s better to read an entire line (e.g. readStringUntil('\n')) and then float val = toFloat() from it. This avoids the “kicking” issue of parseFloat. In any case, the code does clamp the result (constrain(-15,15)), which is good.

VescUart Library: The VescUart UART usage is correct. Calling UART.setCurrent(targetCurrent) sends a VESC RPM/current command, and sendKeepalive() refreshes the link. The SCUBAJET Arduino example for this library shows sendKeepalive() must be called at least once per second
; the code’s 50 ms interval is very conservative (20× more frequent), which is safe. If needed, this could be slowed to e.g. 100–500 ms with little effect.

Timing and Loop: Every loop, the sketch checks Serial.available() and parses a float. If no new data arrives, it does not update targetCurrent (it retains the last value). If the PC app stops sending (or serial disconnects), the motor would continue at the last current until VESC times out (if it does). A suggested improvement is to implement a timeout: e.g. if no new data for >200 ms, set targetCurrent=0 to brake. Also, adding a small filter (e.g. linear ramp or low-pass) could smooth rapid changes from the game for safety.

In summary, the sketch’s use of pins and functions is consistent with Arduino specs
. The use of SoftwareSerial at 19200 baud is at the edge of reliability
, and parseFloat() may mis-handle trailing characters
. We recommend handling the input line-wise and validating the parse result for robust operation.

PC vJoy FFB Listener
The C# program captures FFB data and sends it to the Arduino. Key points:

vJoy Interface: It acquires vJoy device ID 1 and registers a callback with FfbRegisterGenCB(). In the callback, it checks for FFBPType.PT_EFFREP (constant force report) and fills an FFB_EFF_CONSTANT structure. This is correct usage. vJoy’s constant-force magnitude is an integer in the range [-10000, +10000], where ±10000 corresponds to full force
. The code then scales this linearly to ±15 A.

Magnitude Range: The code comment "// Magnitude: -10000 to +10000" is consistent with the vJoy SDK documentation
. (We note that other FFB effect parameters use 0–10000 or -10000–10000 ranges as well.)

Serial Output: The code formats the float as ToString("F2")+"\n". This produces a text string like “12.34\n”. The newline terminator is important – it signals end-of-line. On the Arduino side, parseFloat() will stop at the ‘\n’. It’s generally good to include it. The PC code opens the SerialPort at 115200, 8-N-1 (defaults). The device name (“COM3”) must match the one shown for the Arduino (e.g. “CH340” or “Arduino Uno”).

Threading: The vJoy callback occurs on a background thread invoked by the vJoy driver. Writing to a SerialPort inside this callback may block briefly if the port’s output buffer is full. In practice, sending one short string is very fast. For maximum safety, one could queue the command and write it in the main thread, but given the low rate of FFB updates, this is rarely an issue.

Dependencies: The code requires the vJoy driver (which must be installed and a vJoy Device configured for FFB) and the vJoyInterfaceWrap DLL. It uses njz3’s fork of vJoy for Win10/11 compatibility (as commented in the code). Ensure you install a vJoy version compatible with your OS and that the virtual device has FFB enabled in vJoyConf.

Overall, the C# logic is sound. We would double-check that negative magnitudes produce a signed string (they do), and that the serial buffer on the Arduino side is flushed often enough. The formatting with exactly two decimals is more precision than the VESC likely needs, but fine.

Timing and Latency Analysis
The end-to-end latency is the sum of USB, processing, and serial transmission delays. Rough estimates:

USB Serial (PC↔Uno): At 115200 baud, sending a ~7-byte string (“-15.00\n” is 7 characters) takes about 0.6 ms. USB polling and latency might add a few milliseconds. The PC side (C#) and Arduino parseFloat() incur small overhead. We expect on the order of 1–3 ms from sending on the PC to the Arduino loop receiving.

SoftwareSerial (Uno↔VESC): The Uno then writes to its SoftwareSerial port at 19200 baud. Sending the same 7 chars takes ~3.6 ms. The VescUart library then formats this and sends it over UART to the VESC. The VESC’s processing is essentially immediate, but the one-way UART send itself (20 bits/char at 19200 bps) for say 10 bytes is ~5.2 ms. In total, from Arduino write to VESC reception might be ~5–10 ms.

Keepalive Timing: The code sends sendKeepalive() every 50 ms, i.e. 20 Hz. The SCUBAJET example comments that 1 s intervals are sufficient
, so 50 ms is far more frequent than needed. It guarantees the ESC never sees a connection timeout. One could reduce this (e.g. to 100–200 ms) without risk, freeing CPU.

Jitter: The slowest link is the 19200 baud software serial. Arduino’s SoftwareSerial receives each bit by timed interrupts, so heavy interrupt load could drop bits or cause skew. In normal use (light interrupts), 19200 is usually stable
, but occasional jitter is possible. Using hardware UART or an interrupt-driven library (AltSoftSerial) would reduce jitter.

Game/PC FFB Rate: The overall system reaction is also limited by how frequently the game outputs FFB updates (often a few hundred Hz max) and how quickly vJoy and the C# thread dispatch them. In practice, the pipeline above suggests sub-20 ms latency, which should feel quite responsive.

In summary, no bottleneck appears insurmountable. The 115200 USB link is fast, and the 19200 link is the slowest but still sub-10 ms per packet. Keepalive at 50 ms is more than adequate; even 1 Hz is usually fine
. We estimate total round-trip (game input to motor torque) on the order of ~10–20 ms.

Safety Analysis
Current Limits: The sketch clamps targetCurrent to ±15 A. This matches the configured motor current limit. Since the ESC’s continuous rating is 50 A
, this is conservative. If a fault occurs (over-temperature, over-current, or unplugged motor) the VESC will automatically shut down the output per its protection settings
. One should verify in VESC-Tool that the motor and battery current limits are indeed set to ~15 A (and not left at a high default).

Emergency Stop: There is no explicit E-STOP input in this setup. If the PC app or Arduino crashes or the serial link is lost, the Arduino will simply stop receiving new commands. In that case, without additional code the last command persists until a VESC timeout or hit, which could be unsafe if the wheel is spinning. A recommended improvement is to detect serial timeout on the Arduino (e.g. if Serial.available() is false for >200 ms) and then force targetCurrent=0 to gently brake. This provides a basic “fail-stop.”

VESC Fault Modes: The Flipsky ESC will fault on short-circuit, over-voltage (battery), under-voltage (battery), or motor stalls. The manuals highlight checking “wiring integrity” and “phase wires”
to avoid faults. Ensure the 24 V supply cannot exceed ~60 V (avoid over-voltage). If a fault is triggered, the VESC Tool log or LEDs can indicate the cause.

Wiring and Fusing: All high-current leads (24 V to ESC, and ESC to motor) should be thick and secure. Adding a fuse on the 24 V line (size chosen just above the max expected current, e.g. 20–30 A) is strongly recommended to protect against shorts. The XT60 connector itself is rated high, but the power source should include proper battery protection or a circuit breaker. The ESC’s anti-spark switch (the 16 mm button) should not be powered without its LED as Flipsky warns
.

Thermals: The FSESC 4.20 has an aluminum heatsink, but at 15 A continuous it can still heat up. Run initial tests at low current and check ESC temperature by touch or IR. If it gets very hot (e.g. >60°C), consider adding a fan or further reducing current. The hoverboard motor will also heat under load; avoid continuous stall conditions.

Bench Testing: Always test incrementally. We outline a safe bench checklist below. In general, start with no load, verify commands produce no undesired motion, then gradually apply more throttle with safeguards (e.g. have an emergency power cut-off switch handy).

Overall, the clamping and tool limits mean under normal operation the motor cannot draw more than ±15 A. The Flipsky manual includes protective features (over-current, voltage cutoffs)
that should be enabled. With proper fuse, careful testing, and monitoring, the system can be run safely.

Bench Testing Checklist
Verify Wiring (Power Off): Double-check all connections. Battery leads to XT60 on ESC with correct polarity
; phase wires and hall wires to motor as above
. Ensure good solder/jumper connections. Install an appropriate fuse on the 24 V line.
Computer–Arduino Check: Connect the Uno to PC via USB. In Arduino IDE or serial monitor, upload a test sketch that echoes received serial text. Open the serial monitor, send sample values (e.g. 5.0\n) and confirm the Arduino receives and displays them. This tests the PC–USB–Arduino link at 115200 baud.
ESC Connectivity (USB): With no motor connected, plug the ESC in via USB. Launch VESC Tool and confirm it detects the ESC (USB). Check that the anti-spark switch is on. Verify you see ESC voltage reading (~24 V) and can write settings (e.g., change LED colors).
Motor Detection (ESC): Hook up the motor (phases and halls). In VESC Tool, go to Motor Settings → Detect. Run the FOC detection (Hall) wizard. Ensure it completes without errors and that the Hall sensors are detected. Confirm the motor spins smoothly (thumb on locked rotor) and the direction is correct; if reversed, swap two phase wires. Write settings.
Integration Test (No Load): Connect UNO and ESC (via TTL adapter). Run the PC FFB app (or temporarily just write values like 0\n) and observe Arduino prints/clamps correctly. Ensure sendKeepalive() is being sent (use a scope or LED toggle if desired). In VESC Tool’s Real-time data, monitor AppTarget or current to see if the ESC is receiving commands (you can enable Real-time currents). Without the motor on, send small currents (e.g. 1 A) and ensure the ESC response is as expected (no faults).
Low-Load Spin: Secure the motor so it can spin freely but watchful. With all connections stable, set a low forward current (e.g. 2–5 A via FFB input) and gradually increase while monitoring ESC temperature and current. The wheel should turn. Test reverse (game effect negative). Verify the current limit (~15 A) is respected (monitor with Data Recorder or a clamp meter).
Safety Checks: Confirm that emergency power cutoff (removing supply) stops motor immediately. If any anomaly (strange noise, overheating, ESC fault lights), stop and troubleshoot.
After all tests are successful, the system can be connected under actual conditions.

2. SoftwareSerial dual-instance caveat
ATmega328P SoftwareSerial can only listen on one instance at a time — the last one to call begin() wins. Here, wemosSerial.begin(19200) is called after vescSerial.begin(19200), so wemosSerial is the active listener.

/// Wire format for torque commands. Identical regardless of transport so
/// the receiver MCU does not need to know how the command arrived.
/// Frame: "T,<current>\n" where current is fixed-point 2dp in amps.
/// Includes a single-byte XOR checksum to detect Wi-Fi corruption.
/// Final wire: "T,<current>*<XX>\n" where XX is hex of XOR over "T,<current>".
/// </summary>
public static class UnoProtocol
{
public const string CommandPrefix = "T";

public static string TorqueCommand(float currentAmps)
{

byte checksum = ComputeXor(body);
return body + "*" + checksum.ToString("X2", CultureInfo.InvariantCulture) + "\n";
}

public static bool TryParse(string frame, out float currentAmps)
{
currentAmps = 0f;
if (string.IsNullOrEmpty(frame)) return false;

string clean = frame.Trim();
int starIdx = clean.LastIndexOf('*');
if (starIdx < 0 || starIdx + 3 > clean.Length) return false;

string body = clean.Substring(0, starIdx);
string checksumHex = clean.Substring(starIdx + 1, 2);

if (!byte.TryParse(checksumHex, NumberStyles.HexNumber, CultureInfo.InvariantCulture, out byte expected))
return false;

if (parts.Length != 2) return false;
if (parts[0] != CommandPrefix) return false;

return float.TryParse(parts[1], NumberStyles.Float, CultureInfo.InvariantCulture, out currentAmps);
}

private static byte ComputeXor(string s)
{
byte x = 0;
for (int i = 0; i < s.Length; i++) x ^= (byte)s[i];
return x;
}

return Math.Max(-p.PositiveSaturation, Math.Min(p.PositiveSaturation, raw));
}

}
